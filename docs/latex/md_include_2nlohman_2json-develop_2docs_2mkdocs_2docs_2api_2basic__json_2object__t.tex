\doxylink{requirements_The}{The} type used to store JSON objects.

\href{https://tools.ietf.org/html/rfc8259}{\texttt{RFC 8259}} describes JSON objects as follows:\+ \begin{quote}
An object is an unordered collection of zero or more name/\+value pairs, where a name is a string and a value is a string, number, boolean, null, object, or array. \end{quote}


To store objects in C++, a type is defined by the template parameters described below.\hypertarget{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2api_2basic__json_2object__t_autotoc_md631}{}\doxysection{\texorpdfstring{Template parameters}{Template parameters}}\label{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2api_2basic__json_2object__t_autotoc_md631}
{\ttfamily Object\+Type} :\+ the container to store objects (e.\+g., {\ttfamily std::\+map} or {\ttfamily std::\+unordered\+\_\+map})

{\ttfamily String\+Type} :\+ the type of the keys or names (e.\+g., {\ttfamily std::\+string}). \doxylink{requirements_The}{The} comparison function {\ttfamily std::\+less\texorpdfstring{$<$}{<}String\+Type\texorpdfstring{$>$}{>}} is used to order elements inside the container.

\doxylink{requirements_The}{The} choice of {\ttfamily object\+\_\+t} influences the behavior of the JSON class. With the default type, objects have the following behavior:\+


\begin{DoxyItemize}
\item When all names are unique, objects will be interoperable in the sense that all software implementations receiving that object will agree on the name-\/value mappings.
\item When the names within an object are not unique, it is unspecified which one of the values for a given key will be chosen. For instance, {\ttfamily \#!json \{"{}key"{}:\+ 2, "{}key"{}:\+ 1\}} could be equal to either {\ttfamily \#!json \{"{}key"{}:\+ 1\}} or {\ttfamily \#!json \{"{}key"{}:\+ 2\}}.
\item Internally, name/\+value pairs are stored in lexicographical order of the names. Objects will also be serialized (see \doxysectlink{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2api_2basic__json_2dump}{{\ttfamily dump}}{0}) in this order. For instance, {\ttfamily \#!json \{"{}b"{}:\+ 1, "{}a"{}:\+ 2\}} and {\ttfamily \#!json \{"{}a"{}:\+ 2, "{}b"{}:\+ 1\}} will be stored and serialized as {\ttfamily \#!json \{"{}a"{}:\+ 2, "{}b"{}:\+ 1\}}.
\item When comparing objects, the order of the name/\+value pairs is irrelevant. This makes objects interoperable in the sense that they will not be affected by these differences. For instance, {\ttfamily \#!json \{"{}b"{}:\+ 1, "{}a"{}:\+ 2\}} and {\ttfamily \#!json \{"{}a"{}:\+ 2, "{}b"{}:\+ 1\}} will be treated as equal.
\end{DoxyItemize}\hypertarget{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2api_2basic__json_2object__t_autotoc_md635}{}\doxysubsubsection{\texorpdfstring{Limits}{Limits}}\label{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2api_2basic__json_2object__t_autotoc_md635}
\href{https://tools.ietf.org/html/rfc8259}{\texttt{RFC 8259}} specifies:\+ \begin{quote}
An implementation may set limits on the maximum depth of nesting. \end{quote}


In this class, the object\textquotesingle{}s limit of nesting is not explicitly constrained. However, a maximum depth of nesting may be introduced by the compiler or runtime environment. \doxylink{class_a}{A} theoretical limit can be queried by calling the \doxysectlink{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2api_2basic__json_2max__size}{{\ttfamily max\+\_\+size}}{0} function of a JSON object.\hypertarget{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2api_2basic__json_2object__t_autotoc_md636}{}\doxysubsubsection{\texorpdfstring{Storage}{Storage}}\label{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2api_2basic__json_2object__t_autotoc_md636}
Objects are stored as pointers in a {\ttfamily \doxylink{classbasic__json}{basic\+\_\+json}} type. That is, for any access to object values, a pointer of type {\ttfamily object\+\_\+t\texorpdfstring{$\ast$}{*}} must be dereferenced.\hypertarget{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2api_2basic__json_2object__t_autotoc_md637}{}\doxysubsubsection{\texorpdfstring{Object key order}{Object key order}}\label{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2api_2basic__json_2object__t_autotoc_md637}
\doxylink{requirements_The}{The} order name/\+value pairs are added to the object are {\itshape not} preserved by the library. Therefore, iterating an object may return name/\+value pairs in a different order than they were originally stored. In fact, keys will be traversed in alphabetical order as {\ttfamily std::\+map} with {\ttfamily std::\+less} is used by default. Please note this behavior conforms to \href{https://tools.ietf.org/html/rfc8259}{\texttt{RFC 8259}}, because any order implements the specified "{}unordered"{} nature of JSON objects.\hypertarget{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2api_2basic__json_2object__t_autotoc_md638}{}\doxysection{\texorpdfstring{Examples}{Examples}}\label{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2api_2basic__json_2object__t_autotoc_md638}
