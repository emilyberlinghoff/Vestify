\hypertarget{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2index_autotoc_md1518}{}\doxysubsection{\texorpdfstring{Behavior}{Behavior}}\label{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2index_autotoc_md1518}
\doxylink{requirements_The}{The} choice of {\ttfamily object\+\_\+t} influences the behavior of the JSON class. With the default type, objects have the following behavior:\+


\begin{DoxyItemize}
\item When all names are unique, objects will be interoperable in the sense that all software implementations receiving that object will agree on the name-\/value mappings.
\item When the names within an object are not unique, it is unspecified which one of the values for a given key will be chosen. For instance, {\ttfamily \#!json \{"{}key"{}:\+ 2, "{}key"{}:\+ 1\}} could be equal to either {\ttfamily \#!json \{"{}key"{}:\+ 1\}} or {\ttfamily \#!json \{"{}key"{}:\+ 2\}}.
\item Internally, name/\+value pairs are stored in lexicographical order of the names. Objects will also be serialized (see {\ttfamily dump}) in this order. For instance, both {\ttfamily \#!json \{"{}b"{}:\+ 1, "{}a"{}:\+ 2\}} and {\ttfamily \#!json \{"{}a"{}:\+ 2, "{}b"{}:\+ 1\}} will be stored and serialized as {\ttfamily \#!json \{"{}a"{}:\+ 2, "{}b"{}:\+ 1\}}.
\item When comparing objects, the order of the name/\+value pairs is irrelevant. This makes objects interoperable in the sense that they will not be affected by these differences. For instance, {\ttfamily \#!json \{"{}b"{}:\+ 1, "{}a"{}:\+ 2\}} and {\ttfamily \#!json \{"{}a"{}:\+ 2, "{}b"{}:\+ 1\}} will be treated as equal.
\end{DoxyItemize}\hypertarget{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2index_autotoc_md1519}{}\doxysubsection{\texorpdfstring{Key order}{Key order}}\label{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2index_autotoc_md1519}
\doxylink{requirements_The}{The} order name/\+value pairs are added to the object are {\itshape not} preserved by the library. Therefore, iterating an object may return name/\+value pairs in a different order than they were originally stored. In fact, keys will be traversed in alphabetical order as {\ttfamily std::\+map} with {\ttfamily std::\+less} is used by default. Please note this behavior conforms to \href{https://tools.ietf.org/html/rfc8259}{\texttt{RFC 8259}}, because any order implements the specified "{}unordered"{} nature of JSON objects.\hypertarget{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2index_autotoc_md1520}{}\doxysubsection{\texorpdfstring{Limits}{Limits}}\label{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2index_autotoc_md1520}
\href{https://tools.ietf.org/html/rfc8259}{\texttt{RFC 8259}} specifies:\+

\begin{quote}
An implementation may set limits on the maximum depth of nesting. \end{quote}


In this class, the object\textquotesingle{}s limit of nesting is not explicitly constrained. However, a maximum depth of nesting may be introduced by the compiler or runtime environment. \doxylink{class_a}{A} theoretical limit can be queried by calling the {\ttfamily max\+\_\+size} function of a JSON object.\hypertarget{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2index_autotoc_md1521}{}\doxysubsection{\texorpdfstring{Storage}{Storage}}\label{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2index_autotoc_md1521}
Objects are stored as pointers in a {\ttfamily \doxylink{classbasic__json}{basic\+\_\+json}} type. That is, for any access to object values, a pointer of type {\ttfamily object\+\_\+t\texorpdfstring{$\ast$}{*}} must be dereferenced.\hypertarget{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2index_autotoc_md1522}{}\doxysection{\texorpdfstring{Arrays}{Arrays}}\label{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2index_autotoc_md1522}

\href{https://tools.ietf.org/html/rfc8259}{\texttt{RFC 8259}} describes JSON strings as follows:\+

\begin{quote}
\doxylink{class_a}{A} string is a sequence of zero or more Unicode characters. \end{quote}


Unicode values are split by the JSON class into byte-\/sized characters during deserialization.\hypertarget{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2index_autotoc_md1527}{}\doxysubsection{\texorpdfstring{Default type}{Default type}}\label{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2index_autotoc_md1527}
With the default values for {\itshape String\+Type} ({\ttfamily std::\+string}), the default value for {\ttfamily string\+\_\+t} is {\ttfamily \#!cpp std::\+string}.\hypertarget{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2index_autotoc_md1528}{}\doxysubsection{\texorpdfstring{Encoding}{Encoding}}\label{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2index_autotoc_md1528}
Strings are stored in UTF-\/8 encoding. Therefore, functions like {\ttfamily std::\+string::\+size()} or {\ttfamily std::\+string::\+length()} return the number of {\bfseries{bytes}} in the string rather than the number of characters or glyphs.\hypertarget{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2index_autotoc_md1529}{}\doxysubsection{\texorpdfstring{String comparison}{String comparison}}\label{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2index_autotoc_md1529}
\href{https://tools.ietf.org/html/rfc8259}{\texttt{RFC 8259}} states:\+

\begin{quote}
Software implementations are typically required to test names of object members for equality. Implementations that transform the textual representation into sequences of Unicode code units and then perform the comparison numerically, code unit by code unit are interoperable in the sense that implementations will agree in all cases on equality or inequality of two strings. For example, implementations that compare strings with escaped characters unconverted may incorrectly find that {\ttfamily "{}a\textbackslash{}\textbackslash{}b"{}} and {\ttfamily "{}a\textbackslash{}u005\+Cb"{}} are not equal. \end{quote}


This implementation is interoperable as it does compare strings code unit by code unit.\hypertarget{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2index_autotoc_md1530}{}\doxysubsection{\texorpdfstring{Storage}{Storage}}\label{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2index_autotoc_md1530}
String values are stored as pointers in a {\ttfamily \doxylink{classbasic__json}{basic\+\_\+json}} type. That is, for any access to string values, a pointer of type {\ttfamily string\+\_\+t\texorpdfstring{$\ast$}{*}} must be dereferenced.\hypertarget{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2index_autotoc_md1531}{}\doxysection{\texorpdfstring{Booleans}{Booleans}}\label{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2index_autotoc_md1531}
\href{https://tools.ietf.org/html/rfc8259}{\texttt{RFC 8259}} implicitly describes a boolean as a type which differentiates the two literals {\ttfamily true} and {\ttfamily false}.\hypertarget{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2index_autotoc_md1532}{}\doxysubsection{\texorpdfstring{Default type}{Default type}}\label{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2index_autotoc_md1532}
With the default values for {\itshape Boolean\+Type} ({\ttfamily \#!cpp bool}), the default value for {\ttfamily boolean\+\_\+t} is {\ttfamily \#!cpp bool}.\hypertarget{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2index_autotoc_md1533}{}\doxysubsection{\texorpdfstring{Storage}{Storage}}\label{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2index_autotoc_md1533}
Boolean values are stored directly inside a {\ttfamily \doxylink{classbasic__json}{basic\+\_\+json}} type.\hypertarget{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2index_autotoc_md1534}{}\doxysection{\texorpdfstring{Numbers}{Numbers}}\label{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2index_autotoc_md1534}
See the \doxysectlink{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2number__handling}{number handling}{0} article for a detailed discussion on how numbers are handled by this library.

\href{https://tools.ietf.org/html/rfc8259}{\texttt{RFC 8259}} describes numbers as follows:\+

\begin{quote}
\doxylink{requirements_The}{The} representation of numbers is similar to that used in most programming languages. \doxylink{class_a}{A} number is represented in base 10 using decimal digits. It contains an integer component that may be prefixed with an optional minus sign, which may be followed by a fraction part and/\+or an exponent part. Leading zeros are not allowed. (...) Numeric values that cannot be represented in the grammar below (such as Infinity and NaN) are not permitted. \end{quote}


This description includes both integer and floating-\/point numbers. However, C++ allows more precise storage if it is known whether the number is a signed integer, an unsigned integer, or a floating-\/point number. Therefore, three different types, {\ttfamily number\+\_\+integer\+\_\+t}, {\ttfamily number\+\_\+unsigned\+\_\+t}, and {\ttfamily number\+\_\+float\+\_\+t} are used.\hypertarget{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2index_autotoc_md1535}{}\doxysubsection{\texorpdfstring{Default types}{Default types}}\label{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2index_autotoc_md1535}
With the default values for {\itshape Number\+Integer\+Type} ({\ttfamily std::\+int64\+\_\+t}), the default value for {\ttfamily number\+\_\+integer\+\_\+t} is {\ttfamily std::\+int64\+\_\+t}. With the default values for {\itshape Number\+Unsigned\+Type} ({\ttfamily std::\+uint64\+\_\+t}), the default value for {\ttfamily number\+\_\+unsigned\+\_\+t} is {\ttfamily std::\+uint64\+\_\+t}. With the default values for {\itshape Number\+Float\+Type} ({\ttfamily \#!cpp double}), the default value for {\ttfamily number\+\_\+float\+\_\+t} is {\ttfamily \#!cpp double}.\hypertarget{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2index_autotoc_md1536}{}\doxysubsection{\texorpdfstring{Default behavior}{Default behavior}}\label{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2index_autotoc_md1536}

\begin{DoxyItemize}
\item \doxylink{requirements_The}{The} restrictions about leading zeros are not enforced in C++. Instead, leading zeros in integer literals lead to an interpretation as an octal number. Internally, the value will be stored as a decimal number. For instance, the C++ integer literal {\ttfamily \#!c 010} will be serialized to {\ttfamily \#!c 8}. During deserialization, leading zeros yield an error.
\item Not-\/a-\/number (NaN) values will be serialized to {\ttfamily \#!json null}.
\end{DoxyItemize}\hypertarget{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2index_autotoc_md1537}{}\doxysubsection{\texorpdfstring{Limits}{Limits}}\label{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2types_2index_autotoc_md1537}
\href{https://tools.ietf.org/html/rfc8259}{\texttt{RFC 8259}} specifies:\+

\begin{quote}
An implementation may set limits on the range and precision of numbers. \end{quote}


When the default type is used, the maximal integer number that can be stored is {\ttfamily \#!c 9223372036854775807} ({\ttfamily INT64\+\_\+\+MAX}) and the minimal integer number that can be stored is {\ttfamily \#!c -\/9223372036854775808} ({\ttfamily INT64\+\_\+\+MIN}). Integer numbers that are out of range will yield over/\+underflow when used in a constructor. During deserialization, too large or small integer numbers will automatically be stored as {\ttfamily number\+\_\+unsigned\+\_\+t} or {\ttfamily number\+\_\+float\+\_\+t}.
