There are myriads of \href{https://json.org}{\texttt{JSON}} libraries out there, and each may even have its reason to exist. Our class had these design goals:\+


\begin{DoxyItemize}
\item {\bfseries{Intuitive syntax}}. In languages such as Python, JSON feels like a first-\/class data type. We used all the operator magic of modern C++ to achieve the same feeling in your code. Check out the examples below and you\textquotesingle{}ll know what I mean.
\item {\bfseries{Trivial integration}}. Our whole code consists of a single header file \href{https://github.com/nlohmann/json/blob/develop/single_include/nlohmann/json.hpp}{\texttt{{\ttfamily json.\+hpp}}}. That\textquotesingle{}s it. No library, no subproject, no dependencies, no complex build system. \doxylink{requirements_The}{The} class is written in vanilla C++11. All in all, everything should require no adjustment of your compiler flags or project settings. \doxylink{requirements_The}{The} library is also included in all popular \href{https://json.nlohmann.me/integration/package_managers/}{\texttt{package managers}}.
\item {\bfseries{Serious testing}}. Our code is heavily \href{https://github.com/nlohmann/json/tree/develop/tests/src}{\texttt{unit-\/tested}} and covers \href{https://coveralls.io/r/nlohmann/json}{\texttt{100\%}} of the code, including all exceptional behavior. Furthermore, we checked with \href{https://valgrind.org}{\texttt{Valgrind}} and the \href{https://clang.llvm.org/docs/index.html}{\texttt{Clang Sanitizers}} that there are no memory leaks. \href{https://github.com/google/oss-fuzz/tree/master/projects/json}{\texttt{Google OSS-\/\+Fuzz}} additionally runs fuzz tests against all parsers 24/\+7, effectively executing billions of tests so far. To maintain high quality, the project is following the \href{https://bestpractices.coreinfrastructure.org/projects/289}{\texttt{Core Infrastructure Initiative (CII) best practices}}. See the \href{https://json.nlohmann.me/community/quality_assurance}{\texttt{quality assurance}} overview documentation.
\end{DoxyItemize}

Other aspects were not so important to us:\+


\begin{DoxyItemize}
\item {\bfseries{Memory efficiency}}. Each JSON object has an overhead of one pointer (the maximal size of a union) and one enumeration element (1 byte). \doxylink{requirements_The}{The} default generalization uses the following C++ data types:\+ {\ttfamily std::\+string} for strings, {\ttfamily int64\+\_\+t}, {\ttfamily uint64\+\_\+t} or {\ttfamily double} for numbers, {\ttfamily std::\+map} for objects, {\ttfamily std::\+vector} for arrays, and {\ttfamily bool} for Booleans. However, you can template the generalized class {\ttfamily \doxylink{classbasic__json}{basic\+\_\+json}} to your needs.
\item {\bfseries{Speed}}. There are certainly \href{https://github.com/miloyip/nativejson-benchmark\#parsing-time}{\texttt{faster JSON libraries}} out there. However, if your goal is to speed up your development by adding JSON support with a single header, then this library is the way to go. If you know how to use a {\ttfamily std::\+vector} or {\ttfamily std::\+map}, you are already set.
\end{DoxyItemize}

{\ttfamily }\hypertarget{dir_66db2a402a8a4d6ef4c9b6ec154914f0_autotoc_md1935}{}\doxysubsubsubsection{\texorpdfstring{{\ttfamily Trailing commas}}{{\ttfamily Trailing commas}}}\label{dir_66db2a402a8a4d6ef4c9b6ec154914f0_autotoc_md1935}

{\ttfamily \doxylink{requirements_The}{The} JSON specification does not allow trailing commas in arrays and objects, and hence this library is treating them as parsing errors by default.}

{\ttfamily Like comments, you can set parameter {\ttfamily ignore\+\_\+trailing\+\_\+commas} to true in the {\ttfamily parse} function to ignore trailing commas in arrays and objects. Note that a single comma as the only content of the array or object ({\ttfamily \+[,]\+} or {\ttfamily \{,\}}) is not allowed, and multiple trailing commas ({\ttfamily \+[1,,]\+}) are not allowed either.}

{\ttfamily This library does not add trailing commas when serializing JSON data.}

{\ttfamily For more information, see \href{https://nigeltao.github.io/blog/2021/json-with-commas-comments.html}{\texttt{JSON With Commas and Comments (JWCC)}}.}

{\ttfamily }\hypertarget{dir_66db2a402a8a4d6ef4c9b6ec154914f0_autotoc_md1936}{}\doxysubsubsubsection{\texorpdfstring{{\ttfamily Order of object keys}}{{\ttfamily Order of object keys}}}\label{dir_66db2a402a8a4d6ef4c9b6ec154914f0_autotoc_md1936}

{\ttfamily By default, the library does not preserve the {\bfseries{insertion order of object elements}}. This is standards-\/compliant, as the \href{https://tools.ietf.org/html/rfc8259.html}{\texttt{JSON standard}} defines objects as "{}an unordered collection of zero or more name/\+value pairs"{}.}

{\ttfamily If you do want to preserve the insertion order, you can try the type \href{https://github.com/nlohmann/json/issues/2179}{\texttt{{\ttfamily nlohmann::\+ordered\+\_\+json}}}. Alternatively, you can use a more sophisticated ordered map like \href{https://github.com/Tessil/ordered-map}{\texttt{{\ttfamily tsl::\+ordered\+\_\+map}}} (\href{https://github.com/nlohmann/json/issues/546\#issuecomment-304447518}{\texttt{integration}}) or \href{https://github.com/nlohmann/fifo_map}{\texttt{{\ttfamily nlohmann::\+fifo\+\_\+map}}} (\href{https://github.com/nlohmann/json/issues/485\#issuecomment-333652309}{\texttt{integration}}).}

{\ttfamily See the \href{https://json.nlohmann.me/features/object_order/}{\texttt{{\bfseries{documentation on object order}}}} for more information.}

{\ttfamily We checked with Valgrind and the Address Sanitizer (ASAN) that there are no memory leaks.}

{\ttfamily If you find that a parsing program with this library does not release memory, please consider the following case, and it may be unrelated to this library.}

{\ttfamily {\bfseries{Your program is compiled with glibc.}} There is a tunable threshold that glibc uses to decide whether to actually return memory to the system or whether to cache it for later reuse. If in your program you make lots of small allocations and those small allocations are not a contiguous block and are presumably below the threshold, then they will not get returned to the OS. Here is a related issue \href{https://github.com/nlohmann/json/issues/1924}{\texttt{\#1924}}.}
