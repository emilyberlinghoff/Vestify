That\textquotesingle{}s all! When calling the {\ttfamily json} constructor with your type, your custom {\ttfamily to\+\_\+json} method will be automatically called. Likewise, when calling {\ttfamily get\texorpdfstring{$<$}{<}your\+\_\+type\texorpdfstring{$>$}{>}()} or {\ttfamily get\+\_\+to(your\+\_\+type\&)}, the {\ttfamily from\+\_\+json} method will be called.

Some important things:\+


\begin{DoxyItemize}
\item Those methods {\bfseries{MUST}} be in your type\textquotesingle{}s namespace (which can be the global namespace), or the library will not be able to locate them (in this example, they are in namespace {\ttfamily ns}, where {\ttfamily person} is defined).
\item Those methods {\bfseries{MUST}} be available (e.\+g., proper headers must be included) everywhere you use these conversions. Look at \href{https://github.com/nlohmann/json/issues/1108}{\texttt{\#1108}} for errors that may occur otherwise.
\item When using {\ttfamily get\texorpdfstring{$<$}{<}your\+\_\+type\texorpdfstring{$>$}{>}()}, {\ttfamily your\+\_\+type} {\bfseries{MUST}} be \href{https://en.cppreference.com/w/cpp/named_req/DefaultConstructible}{\texttt{Default\+Constructible}}. (There is a way to bypass this requirement described later.)
\item In function {\ttfamily from\+\_\+json}, use function \doxysectlink{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2api_2basic__json_2at}{{\ttfamily at()}}{0} to access the object values rather than {\ttfamily operator\+[]\+}. In case a key does not exist, {\ttfamily at} throws an exception that you can handle, whereas {\ttfamily operator\+[]\+} exhibits undefined behavior.
\item You do not need to add serializers or deserializers for STL types like {\ttfamily std::\+vector}:\+ the library already implements these.
\end{DoxyItemize}\hypertarget{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2arbitrary__types_autotoc_md1384}{}\doxysection{\texorpdfstring{Simplify your life with macros}{Simplify your life with macros}}\label{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2features_2arbitrary__types_autotoc_md1384}
If you just want to serialize/\+deserialize some structs, the {\ttfamily to\+\_\+json}/\+{\ttfamily from\+\_\+json} functions can be a lot of boilerplate.

There are six macros to make your life easier as long as you (1) want to use a JSON object as serialization and (2) want to use the member variable names as object keys in that object:\+


\begin{DoxyItemize}
\item \doxysectlink{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2api_2macros_2nlohmann__define__type__non__intrusive}{{\ttfamily NLOHMANN\+\_\+\+DEFINE\+\_\+\+TYPE\+\_\+\+NON\+\_\+\+INTRUSIVE(name, member1, member2, ...)}}{0} is to be defined inside the namespace of the class/\+struct to create code for. It will throw an exception in {\ttfamily from\+\_\+json()} due to a missing value in the JSON object.
\item \doxysectlink{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2api_2macros_2nlohmann__define__type__non__intrusive}{{\ttfamily NLOHMANN\+\_\+\+DEFINE\+\_\+\+TYPE\+\_\+\+NON\+\_\+\+INTRUSIVE\+\_\+\+WITH\+\_\+\+DEFAULT(name, member1, member2, ...)}}{0} is to be defined inside the namespace of the class/\+struct to create code for. It will not throw an exception in {\ttfamily from\+\_\+json()} due to a missing value in the JSON object, but fills in values from an object which is default-\/constructed by the type.
\item \doxysectlink{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2api_2macros_2nlohmann__define__type__non__intrusive}{{\ttfamily NLOHMANN\+\_\+\+DEFINE\+\_\+\+TYPE\+\_\+\+NON\+\_\+\+INTRUSIVE\+\_\+\+ONLY\+\_\+\+SERIALIZE(name, member1, member2, ...)}}{0} is to be defined inside the namespace of the class/\+struct to create code for. It does not define a {\ttfamily from\+\_\+json()} function which is needed in case the type does not have a default constructor.
\item \doxysectlink{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2api_2macros_2nlohmann__define__type__intrusive}{{\ttfamily NLOHMANN\+\_\+\+DEFINE\+\_\+\+TYPE\+\_\+\+INTRUSIVE(name, member1, member2, ...)}}{0} is to be defined inside the class/\+struct to create code for. This macro can also access private members. It will throw an exception in {\ttfamily from\+\_\+json()} due to a missing value in the JSON object.
\item \doxysectlink{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2api_2macros_2nlohmann__define__type__intrusive}{{\ttfamily NLOHMANN\+\_\+\+DEFINE\+\_\+\+TYPE\+\_\+\+INTRUSIVE\+\_\+\+WITH\+\_\+\+DEFAULT(name, member1, member2, ...)}}{0} is to be defined inside the class/\+struct to create code for. This macro can also access private members. It will not throw an exception in {\ttfamily from\+\_\+json()} due to a missing value in the JSON object, but fills in values from an object which is default-\/constructed by the type.
\item \doxysectlink{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2api_2macros_2nlohmann__define__type__intrusive}{{\ttfamily NLOHMANN\+\_\+\+DEFINE\+\_\+\+TYPE\+\_\+\+INTRUSIVE\+\_\+\+ONLY\+\_\+\+SERIALIZE(name, member1, member2, ...)}}{0} is to be defined inside the class/\+struct to create code for. This macro can also access private members. It does not define a {\ttfamily from\+\_\+json()} function which is needed in case the type does not have a default constructor.
\end{DoxyItemize}

!!! info "{}\+Implementation limits"{} \begin{DoxyVerb}- The current macro implementations are limited to at most 64 member variables. If you want to serialize/deserialize
  types with more than 64 member variables, you need to define the `to_json`/`from_json` functions manually.
\end{DoxyVerb}


??? example \begin{DoxyVerb}The `to_json`/`from_json` functions for the `person` struct above can be created with:

```cpp
namespace ns {
    NLOHMANN_DEFINE_TYPE_NON_INTRUSIVE(person, name, address, age)
}
```

Here is an example with private members, where `NLOHMANN_DEFINE_TYPE_INTRUSIVE` is needed:

```cpp
namespace ns {
    class address {
      private:
        std::string street;
        int housenumber;
        int postcode;

      public:
        NLOHMANN_DEFINE_TYPE_INTRUSIVE(address, street, housenumber, postcode)
    };
}
```
\end{DoxyVerb}

\doxylink{requirements_The}{The} library uses {\bfseries{JSON Serializers}} to convert types to JSON. \doxylink{requirements_The}{The} default serializer for {\ttfamily \doxylink{namespacenlohmann_a8ba8c5483f73a36e716558571ff97d57}{nlohmann::\+json}} is {\ttfamily \doxylink{structnlohmann_1_1adl__serializer}{nlohmann::\+adl\+\_\+serializer}} (ADL means \href{https://en.cppreference.com/w/cpp/language/adl}{\texttt{Argument-\/\+Dependent Lookup}}).

Yes. You might want to take a look at \href{https://github.com/nlohmann/json/blob/develop/tests/src/unit-udt.cpp}{\texttt{{\ttfamily unit-\/udt.\+cpp}}} in the test suite, to see a few examples.

If you write your own serializer, you will need to do a few things:\+


\begin{DoxyItemize}
\item use a different {\ttfamily \doxylink{classbasic__json}{basic\+\_\+json}} alias than {\ttfamily \doxylink{namespacenlohmann_a8ba8c5483f73a36e716558571ff97d57}{nlohmann::\+json}} (the last template parameter of {\ttfamily \doxylink{classbasic__json}{basic\+\_\+json}} is the {\ttfamily JSONSerializer})
\item use your {\ttfamily \doxylink{classbasic__json}{basic\+\_\+json}} alias (or a template parameter) in all your {\ttfamily to\+\_\+json}/\+{\ttfamily from\+\_\+json} methods
\item use {\ttfamily nlohmann::\+to\+\_\+json} and {\ttfamily nlohmann::\+from\+\_\+json} when you need ADL
\end{DoxyItemize}
