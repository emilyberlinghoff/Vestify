!!! question \begin{DoxyVerb}Why does

```cpp
json j{true};
```

and

```cpp
json j(true);
```

yield different results (`#!json [true]` vs. `#!json true`)?
\end{DoxyVerb}


This is a known issue, and -- even worse -- the behavior differs between GCC and Clang. \doxylink{requirements_The}{The} "{}culprit"{} for this is the library\textquotesingle{}s constructor overloads for initializer lists to allow syntax like


\begin{DoxyCode}{0}
\DoxyCodeLine{json\ \doxymbox{\hyperlink{classarray}{array}}\ =\ \{1,\ 2,\ 3,\ 4\};}

\end{DoxyCode}


for arrays and


\begin{DoxyCode}{0}
\DoxyCodeLine{json\ \textcolor{keywordtype}{object}\ =\ \{\{\textcolor{stringliteral}{"{}one"{}},\ 1\},\ \{\textcolor{stringliteral}{"{}two"{}},\ 2\}\};\ }

\end{DoxyCode}


for objects.

!!! tip \begin{DoxyVerb}To avoid any confusion and ensure portable code, **do not** use brace initialization with the types `basic_json`, `json`, or `ordered_json` unless you want to create an object or array as shown in the examples above.
\end{DoxyVerb}

!!! question "{}\+Questions"{} \begin{DoxyVerb}- Why is the parser complaining about a Chinese character?
- Does the library support Unicode?
- I get an exception `[json.exception.parse_error.101] parse error at line 1, column 53: syntax error while parsing value - invalid string: ill-formed UTF-8 byte; last read: '"Testé$')"`
\end{DoxyVerb}


\doxylink{requirements_The}{The} library supports {\bfseries{Unicode input}} as follows:\+


\begin{DoxyItemize}
\item Only {\bfseries{UTF-\/8}} encoded input is supported, which is the default encoding for JSON, according to \href{https://tools.ietf.org/html/rfc8259.html\#section-8.1}{\texttt{RFC 8259}}.
\item {\ttfamily std::\+u16string} and {\ttfamily std::\+u32string} can be parsed, assuming UTF-\/16 and UTF-\/32 encoding, respectively. These encodings are not supported when reading from files or other input containers.
\item Other encodings such as Latin-\/1 or ISO 8859-\/1 are {\bfseries{not}} supported and will yield parse or serialization errors.
\item \doxylink{requirements_The}{The} library will not replace \href{http://www.unicode.org/faq/private_use.html\#nonchar1}{\texttt{Unicode noncharacters}}.
\item Invalid surrogates (e.\+g., incomplete pairs such as {\ttfamily \textbackslash{}u\+DEAD}) will yield parse errors.
\item \doxylink{requirements_The}{The} strings stored in the library are UTF-\/8 encoded. When using the default string type ({\ttfamily std::\+string}), note that its length/\+size functions return the number of stored bytes rather than the number of characters or glyphs.
\item When you store strings with different encodings in the library, calling \href{https://nlohmann.github.io/json/classnlohmann_1_1basic__json_a50ec80b02d0f3f51130d4abb5d1cfdc5.html\#a50ec80b02d0f3f51130d4abb5d1cfdc5}{\texttt{{\ttfamily dump()}}} may throw an exception unless {\ttfamily json::\+error\+\_\+handler\+\_\+t::\+replace} or {\ttfamily json::\+error\+\_\+handler\+\_\+t::\+ignore} are used as error handlers.
\end{DoxyItemize}

In most cases, the parser is right to complain, because the input is not UTF-\/8 encoded. This is especially true for Microsoft Windows, where Latin-\/1 or ISO 8859-\/1 is often the standard encoding.\hypertarget{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2home_2faq_autotoc_md1649}{}\doxysubsection{\texorpdfstring{Wide string handling}{Wide string handling}}\label{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2home_2faq_autotoc_md1649}
!!! question \begin{DoxyVerb}Why are wide strings (e.g., `std::wstring`) dumped as arrays of numbers?
\end{DoxyVerb}


As described above, the library assumes UTF-\/8 as encoding. To store a wide string, you need to change the encoding.

!!! example \begin{DoxyVerb}```cpp
#include <codecvt> // codecvt_utf8
#include <locale>  // wstring_convert

// encoding function
std::string to_utf8(std::wstring& wide_string)
{
    static std::wstring_convert<std::codecvt_utf8<wchar_t>> utf8_conv;
    return utf8_conv.to_bytes(wide_string);
}

json j;
std::wstring ws = L"車B1234 こんにちは";

j["original"] = ws;
j["encoded"] = to_utf8(ws);

std::cout << j << std::endl;
```

The result is:

```json
{
  "encoded": "車B1234 こんにちは",
  "original": [36554, 66, 49, 50, 51, 52, 32, 12371, 12435, 12395, 12385, 12399]
}
```
\end{DoxyVerb}

!!! question \begin{DoxyVerb}- It seems that precision is lost when serializing a double.
- Can I change the precision for floating-point serialization?
\end{DoxyVerb}


\doxylink{requirements_The}{The} library uses {\ttfamily std::\+numeric\+\_\+limits\texorpdfstring{$<$}{<}number\+\_\+float\+\_\+t\texorpdfstring{$>$}{>}::digits10} (15 for IEEE {\ttfamily double}s) digits for serialization. This value is sufficient to guarantee roundtripping. If one uses more than this number of digits of precision, then string -\/\texorpdfstring{$>$}{>} value -\/\texorpdfstring{$>$}{>} string is not guaranteed to round-\/trip.

!!! quote "{}\+[cppreference.\+com]\+(https:\+/\+/\+en.\+cppreference.\+com/\+w/\+cpp/\+types/\+numeric\+\_\+limits/\+digits10)"{} \begin{DoxyVerb}The value of `std::numeric_limits<T>::digits10` is the number of base-10 digits that can be represented by the type T without change, that is, any number with this many significant decimal digits can be converted to a value of type T and back to decimal form, without change due to rounding or overflow. 
\end{DoxyVerb}


!!! tip \begin{DoxyVerb}The website https://float.exposed gives a good insight into the internal storage of floating-point numbers.
\end{DoxyVerb}

!!! question \begin{DoxyVerb}Why does the code not compile with Android SDK?
\end{DoxyVerb}


Android defaults to using very old compilers and C++ libraries. To fix this, add the following to your {\ttfamily Application.\+mk}. This will switch to the LLVM C++ library, the Clang compiler, and enable C++11 and other features disabled by default.


\begin{DoxyCode}{0}
\DoxyCodeLine{APP\_STL\ :=\ c++\_shared}
\DoxyCodeLine{NDK\_TOOLCHAIN\_VERSION\ :=\ clang3.6}
\DoxyCodeLine{APP\_CPPFLAGS\ +=\ -\/frtti\ -\/fexceptions}

\end{DoxyCode}


\doxylink{requirements_The}{The} code compiles successfully with \href{https://developer.android.com/ndk/index.html?hl=ml}{\texttt{Android NDK}}, Revision 9 -\/ 11 (and possibly later) and \href{https://www.crystax.net/en/android/ndk}{\texttt{CrystaX\textquotesingle{}s Android NDK}} version 10.\hypertarget{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2home_2faq_autotoc_md1657}{}\doxysubsection{\texorpdfstring{Missing STL function}{Missing STL function}}\label{md_include_2nlohman_2json-develop_2docs_2mkdocs_2docs_2home_2faq_autotoc_md1657}
